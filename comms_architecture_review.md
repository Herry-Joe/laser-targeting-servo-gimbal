# 硬件通信方案与系统架构优化评估报告

> 项目：28BYJ-48 微型减速步进电机云台 · 激光打靶系统（K230 视觉板 + STM32F103C8T6）
> 评估范围：K230↔STM32 串口链路、蓝牙/调试遥测、错误处理、延迟与可靠性；以及整体架构、耦合度、性能瓶颈、安全性、可扩展性、代码质量。
> 代码基线：`k230/main.py`（v4.1）、`stm32_tb6612_28byj48_demo/src/main.c`（v4.5）、`stepper.c/.h`、`ssd1306.c/.h`、`oled_debug.c/.h`

## 1. 硬件通信方案现状（链路拓扑）

| 链路 | 波特率 | 方向 | 帧格式 | 频率 | 校验 |
|---|---|---|---|---|---|
| K230(UART2) → STM32(USART3) | 115200 | 单向（控制） | `0x3C 0x3B [XH][XL][YH][YL][FLAG] 0x01 0x01`（9B） | 25 Hz（40 ms） | **无** |
| STM32(USART3) → K230(UART2) | 115200 | 单向（状态回传） | `0x3C 0x3D [YAW][PITCH][STATE][FLAGS][PAD][CKSUM]`（8B） | 2 Hz（500 ms） | 有（和校验） |
| STM32(USART1) → PC/终端 | 115200 | 调试 | 文本 `printf` | 事件触发 | — |
| STM32(USART2) → JDY-31 蓝牙 | 9600 | 遥测 + 命令 | `BT_SendReport` 文本块 + 命令回显 | 0.33 Hz（3 s） | — |
| PC(Web Serial) → JDY-31 | 9600 | 命令 | 单字符 `i/I/P/d` | 人工 | — |

**关键事实**：
- 真正驱动电机的唯一数据通路是 **K230→STM32 的 9 字节控制帧**，但它**没有校验和**，只有固定尾 `0x01 0x01`。
- 反向状态帧（STM32→K230）反而**有**和校验——即“不重要的数据有保护，重要的控制数据无保护”，保护方向**本末倒置**。
- `printf`（`_write`，main.c:532）会**同时向 USART1/USART2/USART3 三路阻塞发送**同一份文本。这是后续大量问题的根源。

## 2. 通信协议评估

### 2.1 协议选择
- 选用**二进制定长帧 + 0x3C 帧头**是合理的（解析快、确定性强），比纯文本 `$cx,cy#` 更适合 115200 实时控制。
- 帧头 `0x3C 0x3B`（坐标）/ `0x3C 0x3D`（状态）区分通道，思路清晰。
- **问题**：控制帧用固定尾 `0x01 0x01` 代替校验，等于把完整性判断交给了“两个特定值的巧合”，在电磁干扰下极易误判。

### 2.2 数据传输效率
- 控制帧 9 B × 25 Hz ≈ **2.25 KB/s @115200**，带宽利用率低（绰绰有余），效率不是瓶颈。
- **真正低效在主控侧**：`printf` 把调试文本也发到三路串口。以 `[Pan] err=…` 为例，每 5 帧（≈5 Hz）一次、约 40–60 B，发到 **9600 蓝牙**需 50–120 ms 阻塞；发到 USART3 又占用 K230 的接收缓冲，与状态帧交错，导致 K230 的 `recv_status()` 经常读不到状态帧（前导被调试文本顶掉，返回 None）。
- 蓝牙本身 9600 是 JDY-31 默认速率，遥测刷新被限速在 3 s，对曲线显示够用，但调试文本塞进去纯属浪费且有害。

### 2.3 错误处理与重传机制
- **无校验**：控制帧任一字节翻转（步进电机 + TB6612 紧邻 UART 线，EMI 显著）会被**静默接受**，误差可达 ±256 px → 电机误动作、抖动、打偏。
- **无序号/无丢帧检测**：多帧在两次主循环间到达时只保留最后一帧（可接受），但丢失或重复无从感知。
- **无重传/无 ACK**：纯 fire-and-forget。对 25 Hz 控制而言，逐帧重传反而增延迟，**不建议加重传**；正确做法是“**校验 + 丢弃非法帧 + 超时安全停**”三件套。
- 已有 `K230_TimeoutCheck`（500 ms 无帧 → 停电机，main.c:905）是良好兜底，但 500 ms 在噪声链路上可能偏长（失步后还跑 0.5 s）。

### 2.4 延迟与可靠性
- 端到端：K230 检测（~12–13 fps 推理）→ 40 ms 发送间隔 → STM32 主循环处理 → 步进机械响应，约 **100–200 ms**，对打靶可接受。
- **抖动来源**：主循环内对 9600 蓝牙的阻塞 `HAL_UART_Transmit`（在 `_write` 与 `BT_SendReport` 中）会卡住主循环，使帧处理周期不恒定 → 控制不稳。这正是上一轮“还有优化空间”的潜在残余根因之一。
- 所有发送均为**轮询阻塞**，无 DMA、无 TX 中断。短帧（K230/状态）影响小；长文本（蓝牙）是瓶颈。

## 3. 项目整体评估

### 3.1 架构合理性
- **分层基本正确**：`stepper` / `ssd1306` / `oled_debug` 已抽成独立模块；`main.c` 负责编排。
- **但 `main.c` 是上帝文件**（~1500 行）：串口初始化、ISR 帧解析、PID、双轴控制、OLED、蓝牙上报、命令解析、云台模式机、按键、舞蹈/扫描全塞在一起。职责过多，难以单独测试。
- 控制环路是**裸机超级循环 + `HAL_GetTick()` 软定时**，无 RTOS，确定性依赖“主循环足够快且不长时间阻塞”——当前被蓝牙阻塞打破。

### 3.2 模块耦合度
- 全局可变状态泛滥：`g_k230_err_x/y`、`g_motor_pan/tilt`、`g_gimbal_mode`、`g_pan_dir_invert` 等跨函数直接读写。
- **亮点**：ISR 中修改的 `g_k230_fstate/fbuf/frame_ready/err_x` 已正确声明 `volatile`（main.c:106–112），无常见的“优化掉共享变量”坑。
- **隐患**：`g_pan_dir_invert` / `g_gimbal_mode` 在命令处理函数里改写、在主循环读取；虽命令解析在**主循环上下文**（USART2 ISR 只缓冲，main.c:1484），故非 ISR 竞争，但协议契约（双端魔数）全靠口头/注释约定，易漂移。

### 3.3 性能瓶颈
1. **9600 蓝牙阻塞发送**（P1，见 2.2/2.4）——主循环抖动主因。
2. **K230 侧 YOLO 推理 ~12 fps**（视觉端，非 MCU），控制 25 Hz 靠插值/EMA 弥补，可接受。
3. OLED 每 200 ms 整屏重绘（已做脏页优化，v4.4），I2C 占用可控。
4. `gc.collect()` 每 15–30 帧（K230）——Python 侧必要开销。

### 3.4 安全性风险
- **开环位置**：28BYJ-48 为步进电机，位置靠“步数累加”估计，无编码器/限位开关回环。一旦失步（机械卡滞、EMI 串入步进线），位置漂移**不可自恢复**，需重新上电归零。
- Tilt 已在 v4.5 加 ±35° 硬限位（`TILT_MAX_STEPS=400`），但 **Pan 轴无同类限位**；水平失步同样危险。
- **蓝牙命令无鉴权**：任何能连上 JDY-31 的设备可发 `i/I` 翻转方向、`d` 进入舞蹈模式。演示环境风险低，但属“可被恶意干扰”面。
- 无看门狗（IWDG）兜底：若主循环跑飞，电机可能停在某个错误状态。

### 3.5 可扩展性
- 加第三轴 / 换视觉板（如贴出的 `cv_lite` 参考方案）/ 改协议，都需大改 `main.c` 与 `k230/main.py` 两处且手动同步魔数。
- 控制帧协议与控制逻辑耦合在同一文件，缺乏独立 `proto` 层。
- 正向：PID、Kalman、Stepper 等算法单元已相对独立，便于复用。

### 3.6 代码质量
- 优点：ISR 变量 `volatile` 处理得当；帧解析有状态机；有超时安全停；Tilt 限位防御性编码。
- 待改进：魔法数（部分已 `#define`，部分仍内联）、`printf` 兼作遥测（职责混淆）、单测缺失、注释与代码偶有不同步（如 `I2C1_Init` 已置空但头注释仍描述硬件 I2C）。

## 4. 优化建议与优先级排序

### P0 — 比赛前必做（可靠性/安全）

**P0-1｜控制帧加校验和（最优先）**
- 位置：K230 `send_coord()`（main.py:184）、STM32 `USART3_IRQHandler`（main.c:1506）。
- 改法：9 字节帧末的 `0x01 0x01` 改为 `0x01 [CRC8]`，或在前 7 字节后追加 1 字节 CRC8（Polynomial 0x07）。STM32 解析时校验失败→丢弃并计数，不更新 `g_k230_err_x`。
- 收益：EMI 下单字节翻转不再驱动电机，直接消除“莫名抖动/打偏”。改动量极小（≈20 行）。

**P0-2｜`printf` 停止扇出到 USART2/USART3**
- 位置：`_write`（main.c:532）。
- 改法：调试文本**仅发 USART1**；蓝牙只走 `BT_SendReport`（已结构化），K230 只走 `K230_SendStatus`。可引入 `dbg_printf()` 宏仅指向 USART1，保留现有 `printf` 调用但截断到非调试串口。
- 收益：消除主循环对 9600 的阻塞（抖动↓↓）、K230 接收不再被调试文本污染、PC 上位机曲线更稳。
- 附带：K230 端 `recv_status` 可稳定收到状态帧。

**P0-3｜Pan 轴加绝对位置硬限位**
- 位置：`K230_ControlAxis`（main.c）的 `pos_limit_steps` 参数；Pan 当前传 `0`（无限制）。
- 改法：仿 Tilt 设 `PAN_MAX_STEPS`（如 600 ≈ ±52°，依机械行程定），上电零点。越界即停+清积分。
- 收益：水平失步后不会无限旋转，系统可恢复。

**P0-4｜加 IWDG 看门狗**
- 主循环定期喂狗；跑飞时复位而非卡死在错误姿态。

### P1 — 应当做（稳定性/延迟）

**P1-1｜控制帧加 1 字节序号**（`g_k230_seq`）
- K230 每帧自增，STM32 记录期望序号；跳号即记一次丢帧（仅统计/调试，不重传）。便于赛后量化链路质量。

**P1-2｜蓝牙/状态发送改非阻塞**
- 用 DMA 或“环形缓冲 + TXE 中断”替代 `HAL_UART_Transmit` 轮询；至少把 `BT_SendReport` 的文本发送移出主循环关键路径（先入队，DMA 后台发）。

**P1-3｜命令通道一致性**
- 现状：USART1 仅回显不解析（main.c:1454），USART2 才解析命令。若评审/演示用串口终端（USART1）会“发命令无效”。建议两路共用同一命令解析，或文档明确“仅蓝牙可发命令”。

**P1-4｜调优超时与死区**
- `K230_FRAME_TIMEOUT_MS=500` 在噪声环境偏长，可降到 200–300 ms（仍远大于 40 ms 帧周期，不会误停）；结合 P0-1 校验后更稳妥。

**P1-5｜方向反转标志做 `volatile` + 原子写**
- `g_pan_dir_invert`/`g_tilt_dir_invert` 虽当前在主循环改，但为防将来在 ISR 改或编译器优化，建议加 `volatile` 并集中到一处写入。

### P2 — 架构级（可维护性/扩展性）

**P2-1｜抽取通信协议层** `k230_proto.c/.h`
- 把帧编/解码、校验、序号从 `main.c` 与 `main.py` 抽出，双端共用同一份常量定义（可用 `.h` + Python 镜像）。降低魔数漂移。

**P2-2｜控制层单元化 + 单测**
- PID / Kalman / 帧解析改为可独立编译、可被主机（PC）单元测试的纯函数；算法正确性不依赖上板。

**P2-3｜考虑轻量调度**
- 引入 Round-Robin 协作调度或 FreeRTOS，把“帧处理 / OLED / 蓝牙发送 / 命令解析”分任务，保证控制帧处理最坏延迟有界。

**P2-4｜文档化协议**
- 在仓库放 `PROTOCOL.md`，列清所有帧格式、波特率、FLAG 位定义，作为 K230 与 STM32 的单一事实来源。

**P2-5｜统一日志分级**
- 定义 `LOG_DEBUG/INFO/WARN`，调试级仅 USART1，关键信息级可进蓝牙/状态帧。彻底去除“printf 既调试又遥测”的混用。

## 6. 给负责人的一句话建议
**先用 P0-1 + P0-2 两处小改动解决“控制帧无校验 + 调试文本阻塞”这两个最高风险点（合计不到半天），它们直接决定打靶的稳定性与可靠性；P0-3/P0-4 补齐安全兜底；P1/P2 作为赛后架构债逐步偿还。** 当前系统功能已闭环，问题不在“能不能打中”，而在“在电磁干扰和长时间运行下是否稳、是否不会失控”。
